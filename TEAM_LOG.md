## Правила ведения

1. **Append-only:** История лога не переписывается. Записи добавляются только снизу (внутри раздела участника).
2. **"Текущая реальность" — исключение:** Этот блок вверху файла можно обновлять для синхронизации общего контекста.
3. **Формат записи строгий:** ID (XX-###), Дата, Статус, Факт/Решение, Границы, Подтверждение, Следующий шаг.
4. **Проверки заканчиваются явно:** Любая проверка должна иметь итог PASS или FAIL (не оставлять "в подвешенном" виде).
5. **"Принято/проверено" со статусом и границами:** Любое утверждение "принято" или "проверено" обязано содержать статус (PASS/FAIL/BACKLOG) и границы (что входит / что не входит).
6. **Одна запись = один факт/решение:** Не смешивать несколько решений в одной записи.
7. **ID не повторяются:** XX = префикс участника (DUM/QA/TR/DEV/NAV), ### = порядковый номер внутри участника.
8. **Ссылки на записи:** Чтобы сослаться на решение другого участника, пишем ID (например: "по NAV-001" или "см. DEV-005").
9. **Короткие формулировки:** Факт/Решение — одна фраза, Границы — 1-2 строки, Следующий шаг — одна команда.
10. **Никаких изменений кода под предлогом лога:** Этот файл — только документация. Код игры/механика/экономика не затрагиваются.
11. **Инвариант “всегда писать в лог”:** После каждого значимого ответа/действия добавлять запись в свой раздел (append-only) с Evidence/Impact; если есть неопределённость — фиксировать как INFO/BLOCKER/FAIL, но запись всё равно обязательна.
