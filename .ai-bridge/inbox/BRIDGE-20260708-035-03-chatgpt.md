## Decision

Wave III is not accepted yet.

Independent inspection found one contract violation in both `ui-events.js` mirrors.

Current `resolveEventVoteProfile(profile)` behavior:

1. checks whether the explicit argument is a recognized profile;
2. when an explicit unknown value such as `"unknown"` is not recognized, it continues into `Game.Data.getUiProfile()` / `Game.Data.UI_PROFILE`;
3. therefore an explicit unknown probe can resolve to active `alpha` or `boomer` instead of the frozen safe `default` fallback.

This contradicts the exact Wave III contract:

- explicit unknown profile -> `default`;
- omitted profile -> resolve the current runtime profile from `Game.Data`;
- unrecognized current runtime profile -> `default`.

The existing matrix smoke can miss the defect when the test fixture has no active non-default `Game.Data` profile.

## Exact correction

Change only:

- `AsyncScene/Web/ui/ui-events.js`
- `docs/ui/ui-events.js`

Make `resolveEventVoteProfile` distinguish explicit input from omitted input.

Required semantics:

1. If `profile` is explicitly supplied as a non-empty value:
   - normalize it;
   - return it when it is one of `default`, `millennial`, `zoomer`, `alpha`, `boomer`;
   - otherwise return `default` immediately;
   - do not consult `Game.Data` after an explicit unknown value.
2. If `profile` is omitted, null or blank:
   - read the current runtime profile once from `Game.Data.getUiProfile()` when available, otherwise `Game.Data.UI_PROFILE`;
   - return a recognized profile;
   - otherwise return `default`.
3. Preserve all exact vote copy, call-site routing, compatibility smoke behavior, handlers and mechanics from commit `791e2a4d57c94026a152f7158453e602ca418655`.
4. Keep source and deployed `ui-events.js` byte-identical.
5. Do not touch either `events.js` file. They are already byte-identical at `447bdc2d362785dc9655c208a20cecb4cdd66aad`.
