# Economy wave 4 — Scope package (for Valera gate)

Цель: зафиксировать **минимальный** и проверяемый scope для Economy wave 4 на основе `AsyncScene/Web/economy update.txt`.

## 1) Что берём из `AsyncScene/Web/economy update.txt`
Берём только “тон как давление” в части, которая **не требует выбора тона** и уже описана как следствие текущей модели:
- Тон определяется уровнем Influence, выбора нет
- Если победитель с **высоким тоном** выигрывает у более слабого → победа должна иметь цену:
  - Influence уменьшается у победителя
  - REP не растёт (или растёт минимально)

Источник: `AsyncScene/Web/economy update.txt`, разделы “1. Тон как давление” и “7.1 battle_end”.

## 2) Что НЕ берём в wave 4 (явно out-of-scope)
- “Считается разница тонов/Influence и применяется REP-эффект: апсет/позор/ноль” — это отдельная, более широкая логика (не закрывается минимальными параметрами).
- “Несоответствие тона (желтый против красного = автопоражение)” — отдельная волна.
- NPC правила “непобедимы для обычного игрока” — отдельная волна.
- Любые UI изменения (`AsyncScene/Web/ui/**`) — отдельный UI gate пакет, если понадобится.
- Любые изменения правил аргументов/победы/поражения — запрещены.
- Массовый cleanup/рефактор — запрещён.

## 3) Минимальная механика (in-scope)
Wave 4 добавляет **только один хук** на `battle_end`:

**W4.1 pressure-on-win (только для локального игрока `me`):**
- В исходе `win` (когда `me` победил):
  - если у `me` **высокий тон** и оппонент **существенно слабее** (по заранее зафиксированному правилу) —
    - списать Influence у `me`
    - ограничить прирост REP для `me` в этом исходе (обнулить или сделать минимальным)

Примечание: Т.к. текущая архитектура экономики в `AsyncScene/Web/conflict/conflict-economy.js` в основном применяет эффекты к `me`, wave 4 **не пытается** “наказывать” оппонента/других игроков (это отдельный scope, если понадобится).

## 4) Какие файлы/модули трогаем (ограничение)
Разрешённый минимум для реализации:
- `AsyncScene/Web/conflict/conflict-economy.js` — точка применения результатов `win/lose/draw` (одно место, без UI)
- (опционально, только для параметров) `AsyncScene/Web/data.js` — хранение чисел/порогов и reasons

Запрещено:
- Любые UI файлы (`AsyncScene/Web/ui/**`)
- `AsyncScene/Web/conflict/conflict-arguments.js` (аргументы)
- Любые изменения `Points` из этого хука (в wave 4 pressure-on-win Points не трогаем)

## 5) Инварианты wave 4
- REP v1.0 инварианты сохраняются:
  - REP меняется через `transferRep`
  - `addRep` не используется в prod-логике (dev-only допускается существующим guard)
  - каждая операция имеет `reason` и `battleId/eventId`
  - операции логируются в `moneyLog`/`moneyLogByBattle` (как уже принято в wave 1–3)
- Points:
  - хук wave 4 **не меняет** Points
- Influence:
  - политика клипа должна быть зафиксирована gate-ом: клип до 0 или допускаем отрицательные (по умолчанию — клип до 0)

## 6) Открытые параметры (должны быть решены gate-ом Валеры)
Чтобы не “додумывать механику” от имени ассистента, значения фиксирует Valera:
- Что считается “высоким тоном”:
  - например `tier ∈ {r,k}` или `influence >= PROGRESSION.unlockInfluence.power`
- Что считается “существенно слабее”:
  - например порог по Influence (`myInf - oppInf >= X`)
  - или порог по tier-разнице (`tierDiff >= Y`)
- `INF_PRESSURE_WIN_COST` (целое число, сколько Influence списывать у победителя)
- Ограничение REP прироста:
  - `REP_PRESSURE_WIN_CAP` (например `0` или `1`)
  - и/или правило “если условие давления сработало — repGain = min(repGain, cap)”
- `reason` строки для аудита (`rg`) и логов:
  - для REP: либо оставить `rep_battle_win` с изменённым amount, либо добавить отдельный reason (решение gate)
  - для Influence: если добавляется лог — как называется reason/тип записи

## 7) Критерии PASS/FAIL для wave 4 (gate + аудит)
### PASS (по фактам)
- В `AsyncScene/Web/conflict/conflict-economy.js` для исхода `win`:
  - при выполнении условия “высокий тон + слабый оппонент”:
    - Influence у `me` уменьшается на `INF_PRESSURE_WIN_COST` согласно политике клипа
    - REP прирост для `me` ограничен согласно `REP_PRESSURE_WIN_CAP`/правилу gate
  - вне условия: поведение win остаётся прежним (кроме неизбежных структурных правок)
- Нет изменений UI файлов
- Нет изменений `Points` из этого хука
- Нет использования `addRep` в prod-контурах

### FAIL
- Любые изменения UI файлов
- Любые изменения `Points` из pressure-on-win
- Использование `addRep` в прод-контурах
- Нет явной фиксации параметров (high tone / weak threshold / cost / cap) в Result gate-задачи
