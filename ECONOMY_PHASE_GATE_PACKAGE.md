## 3) Предлагаемый объём Economy phase (на решение gate)
Ниже — **предложение** объёма для рассмотрения Valera, чтобы экономику можно было внедрять контролируемо.

### 3.1 In-scope (допустить в Economy phase)
**E1. Экономика как “ядро правил и логов” (без UI-маркетинга):**
- Изменения только в экономических контурах: начисления/списания/трансферы, логи, инварианты.
- Разрешено добавлять/изменять код в модулях:
  - `AsyncScene/Web/state.js`
  - `AsyncScene/Web/events.js`
  - `AsyncScene/Web/conflict/conflict-core.js`
  - `AsyncScene/Web/conflict/conflict-economy.js`
  - (опционально) новые файлы внутри `AsyncScene/Web/conflict/` или `AsyncScene/Web/` с явным префиксом `economy-*` для локализации.

**E2. REP v1.0 как “не ломать”:**
- Любые изменения экономики обязаны **не нарушать** инварианты REP v1.0.

**E3. Canon v1.1 как “референс”:**
- Разрешено реализовывать пункты Canon v1.1 только если они:
  - явно описаны в `AsyncScene/Web/economy update.txt`,
  - не требуют “додумываний” механики за пределами текста,
  - имеют проверяемые критерии.

### 3.2 Out-of-scope (запретить в Economy phase)
**X1. Cleanup/рефактор без необходимости:**
- Запрещены массовые переезды/переименования/перестройка модулей “ради красоты”.

**X2. UI/UX обещания как часть экономики:**
- Любые изменения UI/текстов/кнопок, которые “обещают” экономику игроку, выносятся в отдельную фазу (например “Economy UI honesty/display”), чтобы не смешивать механику и UX.
- Исключение (если Valera разрешит): минимальные UI-подписи для предотвращения явной лжи (но это должно быть отдельным gate-пунктом).

**X3. Новые механики не из канона:**
- Запрещено добавлять новые экономические сущности/кнопки/правила, которых нет в источнике `economy update.txt`.

## 4) Критерии gate PASS/FAIL/BACKLOG (предложение)
### PASS (Valera)
Если одновременно:
- Объём работ (in-scope/out-of-scope) чётко определён и проверяем.
- Назначен владелец реализации: **Миша** (см. `AsyncScene/Web/economy update.txt`: “Владелец Economy Canon: Миша”).
- Есть план аудита Димы (read-only) по фактам.
- Указаны затрагиваемые файлы/модули (см. раздел 2.3) и запрет на “додумывания”.

### FAIL (Valera)
Если есть хотя бы одно:
- В пакете смешаны экономика+UI без явного разрешения gate.
- Объём работ расплывчатый (“внедрить всё” без разбиения и критериев).
- Предлагается нарушать REP v1.0 инварианты (или нет явной оговорки “не ломать”).
- Нет владельца реализации/аудита.

### BACKLOG (Valera)
Если:
- Нужны дополнительные артефакты/уточнения из источников (например вытащить конкретные пункты Canon v1.1 в отдельный “spec” список).
- Не готова инфраструктура проверки (например нет согласованного чеклиста аудита или не определены минимальные сценарии).
